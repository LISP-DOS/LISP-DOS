Módulo FAT16

Documentação Técnica do Kernel LISP-DOS

Visão Geral da Arquitetura

O módulo FAT16 (File Allocation Table) do LISP-DOS é a espinha dorsal de persistência de disco do sistema. Ele abstrai blocos físicos brutos LBA de 512 bytes originados do driver ATA PIO Mode e os transforma em uma estrutura hierárquica legível de arquivos do tipo "8.3" nativo do ecossistema DOS.

Layout do Disco

De acordo com o padrão IBM/Microsoft clássico, o disco formatado como FAT16 possui a seguinte topologia linear mapeada em tempo de Boot pelo nosso módulo:

Região LBA Componente Responsabilidade
LBA 0BPB (Bios Parameter Block)Boot Sector e metadados descritivos da partição (sectors per cluster, root dir entries, etc).
fat_sectorTabela FAT (1 e 2)Tabela ligada (Linked-List) de 16-bits que define alocações em formato "Chain" (corrente) para cada arquivo.
root_dir_sectorRoot DirectoryVetor de registros contendo os Nomes (8.3), Tamanhos, Datas e Cluster Inicial (Entry Point).
data_sectorData Area (Clusters)Oceano de dados brutos reais divididos em clusters numerados a partir do Cluster 2.

Estruturas de Dados Internas

A decodificação binária usa o modelo __attribute__((packed)) do GCC para mapear os bytes exatos dos setores da BIOS.

struct fat16_bpb (Boot Sector)

O cabeçalho mágico lido na rotina de boot.

struct fat16_bpb {
    uint8_t jmp[3];
    char oem_name[8];
    uint16_t bytes_per_sector;      // Geralmente 512
    uint8_t sectors_per_cluster;    // Base de alocação de dados
    uint16_t reserved_sectors;      // Offsets iniciais da FAT
    uint8_t fat_count;              // 2 (Primária e Cópia)
    uint16_t root_dir_entries;      // Quantidade de arquivos máx. na raiz
    uint16_t total_sectors_short;
    uint8_t media_descriptor;
    uint16_t sectors_per_fat;       // Tamanho de cada FAT Table
    // ... Parâmetros Secundários ...
} __attribute__((packed));

struct fat16_dir_entry (Directory Entry)

Cada entrada (seja arquivo ou diretório vazio) na raiz consome exatos 32 bytes estáticos.

struct fat16_dir_entry {
    char name[8];                   // "LISP    "
    char ext[3];                    // "TXT"
    uint8_t attributes;             // 0x10 = Directory, 0x20 = Archive
    uint8_t reserved;
    uint8_t create_time_tenth;
    uint16_t create_time;
    uint16_t create_date;
    uint16_t last_access_date;
    uint16_t first_cluster_high;    // Sempre 0 em FAT16 puro
    uint16_t write_time;
    uint16_t write_date;
    uint16_t first_cluster_low;     // Ponteiro de entrada no FAT Chain
    uint32_t file_size;             // Tamanho real legível
} __attribute__((packed));

Ciclo de Vida do Kernel

fat16_init()

Lê fisicamente o setor nulo via driver ATA e extrai toda a topologia para carregar os marcadores de memória estática: fat_sector, root_dir_sector e o data_sector.

Leitura e Cluster Chaining (fat16_read_file)

O processo de cópia de disco obedece a um laço de repetição while(cluster >= 2 && cluster <= 0xFFEF) que varre os setores vinculados a cada cluster. Utiliza o método auxiliar fat16_get_fat_entry(cluster) que decodifica o buffer 16-bits da FAT Table e descobre qual será o próximo cluster de bytes a ser lido, garantindo consistência em arquivos fragmentados. Possui bounds-checking estrito para copiar bytes idênticos ao tamanho do arquivo (evitando o arrasto do Null Padding de 512b do disco para memória Ring 3).

Operações de Gravação (fat16_write_file)

Alocação de FAT Chain: Quando um novo arquivo é criado, o LISP-DOS varre a FAT buscando clusters que tenham valor 0x0000. A cada cluster preenchido, a tabela é atualizada via fat16_set_fat_entry apontando para o próximo elo, ou para 0xFFFF determinando o fim de fluxo EOF (End of File).

Segurança de HD (Rollback): Caso o disco fique cheio durante a operação, a rotina foi refatorada para varrer reversamente os clusters recém-alocados pelo arquivo abortado e limpá-los para 0x0000, prevenindo o temido "Memory Leak" definitivo de HD.

Subdivide a operação em:

  1. Formatação do nome de arquivo ASCII genérico para o estrito formato FAT 8.3 via format_fat_name.
  2. Varredura no root_dir_sector buscando Slots Vazios (0x00) ou Slots Marcados Excluídos (0xE5).
  3. Geração da trilha FAT com alocação sequencial e salvamento do Entry Point.
  4. Transferência real de dados pelo barramento IO utilizando ata_write_sector alinhados ao `bpb.sectors_per_cluster`.

fat16_get_file_size(const char* filename)

Função de alta performance que sonda apenas os headers do Root Directory comparando strings para colher de imediato a variável estática uint32_t file_size, sendo fundamental para o Módulo Syscall provisionar Memory Pools no espaço de usuário para Editores de Texto e o Interpretador LISP.
