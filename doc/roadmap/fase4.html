Fase 4: Ambientes Gráficos e APIs Avançadas

Até aqui, o LISP-DOS fingiu operar num modo de texto simulado. A Fase 4 desacorrenta todo o potencial da placa de vídeo de 32-bits que já está alocada via Framebuffer, expondo-a para aplicações em espaço de usuário desenharem janelas e gráficos.

Parte 4.1: API de Desenho no Kernel

Problema Atual: O driver vga.c contém apenas rotinas para copiar letras minúsculas (fontes) no framebuffer. Não existem rotinas para plotar geometria matemática.

Tarefas:

  • Implementar o algoritmo de Bresenham para desenhar linhas puras sem serrilhamento através do framebuffer de vídeo com void vga_draw_line(x0, y0, x1, y1, color).
  • Adicionar suporte a elipses, retângulos coloridos e preenchimentos (*Fill routines*).
  • Expor as primitivas via banco de Syscalls (ex: sys_draw_rect) e fornecer a resolução máxima detectada da placa (`sys_get_resolution`).

Parte 4.2: Integração com o Interpretador LISP

Problema Atual: O interpretador só é capaz de somar números e exibi-los no modo texto.

Tarefas:

  • Registrar palavras-chave nativas em lisp.c, como as famosas funções da Linguagem Logo original (tartaruga geométrica). Exemplo: (draw-line 0 0 100 100).
  • Permitir que usuários escrevam scripts matemáticos sofisticados (*fractais*, simulações físicas) salvos no FAT16, que ao serem executados pintam imagens gloriosas na tela.

Parte 4.3: Driver de Mouse e UI

Problema Atual: Navegar em modo gráfico requer um dispositivo apontador de hardware.

Tarefas:

  • Utilizando o Barramento xHCI (construído incansavelmente na Fase 2), extrair a classe de descritor `HID Mouse` para reportes de interrupções de movimento relativas (Eixos X, Y e botões).
  • Atualizar uma ponteiro em formato de "Seta Branca" sobre os pixels da tela no Hardware Layer sempre que a interrupção do controlador for ativada e disparar eventos assíncronos (`Event Queues`) para os processos em User Space absorverem cliques.
