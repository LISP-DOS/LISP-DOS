Parte 6.3: Memory Management Unit (MMU) e MMIO

1. O Mapa de Memória do RPi 4

No Intel x86 nós utilizávamos instruções de portas (in e out) para controlar hardware. A arquitetura ARM usa Memory-Mapped I/O (MMIO), onde os dispositivos físicos respondem a gravações feitas em endereços de RAM arbitrários.

No Raspberry Pi 4 (BCM2711), a base periférica foi drasticamente movida. Enquanto nos Raspberry Pi antigos era em 0x3F000000, no RPi 4 a base MMIO começa nos endereços físicos: 0xFE000000.

Para interagir com o UART, System Timer, Interrupt Controller ou Controlador USB, o Kernel precisa escrever matrizes de bits relativos a este deslocamento de memória: BASE + OFFSET.

2. Ativando a MMU (Paging no ARM)

Da mesma forma que construímos a Árvore de Páginas (PML4) no x86_64 usando o registrador CR3, no AArch64 usaremos o Translation Table Base Register (TTBR0/TTBR1).

Nossa missão será re-implementar a abstração do nosso arquivo vmm.c e pmm.c atual.

  • Configurar o Memory Attribute Indirection Register (MAIR_EL1) com as políticas de cache (Memória Normal vs Memória de Dispositivo).
  • Isolamento Crítico: Endereços que apontam para a placa de memória (RAM) recebem cache L1/L2 para extrema velocidade. Endereços que apontam para o MMIO (0xFE000000) obrigatoriamente deverão ser mapeados como "Device Memory - Strongly Ordered" para que a CPU não emita leituras antecipadas que poderiam dessincronizar transistores sensíveis da placa mãe do Raspberry Pi.
  • Habilitar as quebras de barreira de memória via assembly puro (isb, dmb e dsb).
